Now, the idea is that a cron job will run rsyncing over the files every hour, on the hour.¬† For many reasons (in case I get a virus and need to revert back, in case somebody hacks in and does bad stuff, etc, etc) I‚Äôm going to choose to create Snapshots so I can roll back to a previous version.

The convention I want is hourly.HOUR, daily.DAY, weekly.WEEK for up to 7 days and 4 weeks.¬† This also means that once I delete a file, I won‚Äôt recover the space that it took (once a snapshot of the file has been created) in my data pool until the end of the 4 week period.¬† for instance, hourly.0 will be the last hours snapshot, hourly.1 will be the 2nd last hours snapshot, etc.

the following bash script will take care of the desired snapshots.¬† It‚Äôs based on a concept I took from this rolling snapshots made easy post but I like my scripted way of doing rotating snapshots much better.

#!/bin/bash

#print out usage

if [ $# -ne 2 ]

then

echo ‚ÄúUsage: snapshot.sh [snapName] [max]‚Äù

echo ‚Äù¬† eg. snapshot.sh hour 24‚Ä≥

fi

#if the max snapshot already exists, just delete it

if [ `zfs list -t snapshot | grep datastore@$1.$2 | wc -l` -eq 1 ]

then

zfs destroy -r datastore@$1.$2

fi

for ((i=$2-1; i >= 0; i‚Äì)); do

if [ `zfs list -t snapshot | grep datastore@$1.$i | wc -l` -eq 1 ]

then

#this snapshot exists, so we want to move it up one

zfs rename -r datastore@$1.$i @$1.$[$i+1]

fi

done

zfs snapshot -r datastore@$1.0

so with this snapshot beauty in place, lets say I had an existing file and structure in place
